# 简历 AI 优化与评分质量修订设计

## 背景

当前简历 AI 优化和评分存在两个核心问题：

- 优化输出会把大量正文内容塞进 `highlights` 亮点标签，而不是写入工作经历、项目经历的 `description`。
- 工作经历和项目经历常被生成成一整段泛泛描述，缺少“做了什么、怎么做、产生什么结果”的分点表达。
- AI 评分偏宽松，一份工作/项目内容明显不完整、项目描述较弱的简历仍可能得到 80+ 高分。

这些问题的根因是后端提示词过于通用，只要求“保持原有结构、优化内容质量”，没有按模块定义字段职责、写作格式和严格评分 Rubric。

## 目标

本次修订目标是让 AI 优化和评分都遵循同一套模块化简历质量标准：

- 不同模块有不同写作规则。
- 工作经历和项目经历必须分点说明，每点包含动作、技术/方法、结果/指标。
- `highlights` 只能作为短标签，不能承载正文描述。
- 评分必须严格，不因“模块存在”就给高分。
- 低质量简历需要被识别并给出具体改进建议。

## 非目标

- 不重构前端简历数据结构。
- 不新增 `bullets` 或 `points` 数组字段。
- 不大改评分面板 UI。
- 不引入新的 AI Provider 或外部评分服务。

## 模块化优化规则

### basic 基本信息

只允许校正表达和补全求职意向相关短字段。

约束：

- 不生成经历内容。
- 不生成亮点标签。
- 姓名、邮箱、电话、城市、头像等事实字段不得编造或随意改写。

### education 教育经历

用于表达学校、专业、学历、GPA、课程、奖项、在校项目或活动。

约束：

- `highlights` 最多 3 条，每条必须是短标签或短句。
- 不允许把工作职责、系统开发、接口性能等工作/项目内容写进教育经历。
- 不允许在 `highlights` 中生成长段描述。

示例：

```json
{
  "school": "江西财经大学",
  "major": "软件工程",
  "degree": "本科",
  "gpa": "3.6/4.0（专业前15%）",
  "highlights": ["专业前15%", "数据库课程设计", "校级奖学金"]
}
```

### skills 专业技能

用于表达技能分类和熟练度。

约束：

- 只整理技能分类，不生成项目成果。
- 不虚构原始内容中没有出现过的技术栈。
- 避免堆砌“精通、熟练”等主观词，除非项目/工作中有对应体现。

### work 工作经历

核心字段是 `description`。

约束：

- `description` 必须是 HTML 列表：`<ul><li>...</li></ul>`。
- 每条 `<li>` 说明“负责/参与了什么 + 使用什么技术/方法 + 产生什么结果”。
- 建议 3-5 条；内容较少时至少 2 条。
- 不把完整工作描述写进 `highlights`。
- 如果保留 `highlights`，最多 3 条短标签，例如“订单模块”“接口联调”“SQL 优化”。

示例：

```html
<ul>
  <li>负责 CRM 订单查询与状态变更接口开发，基于 Spring Boot + MyBatis-Plus 完成核心 CRUD、参数校验与异常处理，支撑日处理订单量 5000+。</li>
  <li>参与多表关联查询与分页逻辑优化，通过索引梳理和动态 SQL 改造，将常用查询接口响应时间控制在 150ms 内。</li>
</ul>
```

### project 项目经历

核心字段是 `description`。

约束：

- `description` 必须是 HTML 列表。
- 每条 `<li>` 说明“项目职责 + 技术实现 + 结果/价值”。
- 项目经历要突出模块设计、技术方案、个人贡献、上线效果。
- 不写成公司日常工作流水账。
- 不允许只描述平台背景，不说明个人完成了什么。

示例：

```html
<ul>
  <li>负责在线考试系统题目管理与考试提交模块，基于 Spring Boot 设计题目、试卷、答题记录等核心接口，支撑教师创建试卷和学生在线作答流程。</li>
  <li>使用 MyBatis 实现试卷查询、答题记录保存和自动评分逻辑，减少人工阅卷成本，提升考试管理效率。</li>
</ul>
```

### summary 个人简介

输出 2-4 句短摘要，突出岗位方向、核心技术栈、项目/实习亮点和求职匹配度。

约束：

- 不写成项目列表。
- 不堆砌标签。
- 不夸大没有在经历中体现的能力。

## AI 生成与优化统一规则

以下入口必须使用同一套模块标准：

- AI 生成简历。
- 单模块 AI 优化。
- 整份 AI 优化。

要求：

- 返回 JSON 必须保持当前字段结构。
- 不新增无关字段。
- `work.description` 和 `project.description` 必须使用 HTML 列表。
- `highlights` 只保留短标签。
- 事实字段不编造。

## 评分设计

评分由两层组成：结构化预检查 + AI 严格 Rubric。

### 结构化预检查

后端先根据简历结构计算质量信号，供 AI 评分和分数封顶使用。

检查项：

- 基本信息是否缺姓名、邮箱、电话、目标岗位。
- 技能是否为空或只有未分类堆栈。
- 工作经历是否缺少 `description`。
- 项目经历是否缺少职责、技术实现、结果产出。
- `work/project.description` 是否不是 HTML 列表。
- 是否存在大量空字段、占位文本或极短内容。
- 是否存在量化指标，例如 `5000+`、`150ms`、`30%`、`日处理` 等。

### AI 严格 Rubric

评分提示词必须明确：

- 不要鼓励式评分。
- 不要因为模块齐全就给高分。
- 内容空泛、缺少项目细节、缺少量化结果必须扣分。
- 实习/项目只有一句话，不能判定为高质量。
- 技能写了“熟练”但项目没有体现，不能高分。

分数区间：

- 40-60：结构缺失明显，经历内容空泛，无法支持岗位匹配。
- 61-75：结构基本完整，但工作/项目缺少技术细节、个人贡献或量化结果。
- 76-85：内容较完整，能体现岗位能力，但仍有部分模块表达浅或结果不足。
- 86-100：内容完整，项目/工作分点清晰，技术细节具体，结果可量化，岗位匹配明确。

### 分数封顶规则

后端对 AI 返回分数做二次约束：

- 工作经历和项目经历都没有合格分点描述，总分最高 65。
- 项目经历只有一段泛泛介绍，没有个人职责和结果，总分最高 70。
- 大量模块为空或短内容，总分最高 60。
- 目标岗位是 Java 开发，但项目没有体现 Java 后端核心能力，总分最高 75。
- 没有任何量化结果，总分最高 78。
- `description` 不是列表结构，专业性最高 75。

### 建议输出

评分建议必须指出具体模块和具体问题。

示例：

```text
项目经历“在线考试系统”只有平台介绍，缺少你具体负责的模块、接口设计、数据库表设计、性能指标和上线结果，建议改为 3-5 条分点描述。
```

禁止输出泛泛建议：

```text
项目经验丰富，建议进一步优化。
```

## 后处理设计

AI 返回后做轻量规范化：

- `education.highlights`、`work.highlights`、`project.highlights` 最多保留 3 条。
- 单条 `highlight` 超过长度阈值时丢弃，避免正文进入标签。
- `work/project.description` 如果是普通长文本，尽量转换为 `<ul><li>...</li></ul>`。
- 去除超出当前模块职责的大量无关字段。

## 影响范围

后端：

- `ResumeAiServiceImpl`：提示词、评分预检查、评分封顶、优化结果后处理。
- 可能补充私有辅助方法，不改变 Controller API。

前端：

- 暂不大改评分面板。
- 当前 `FormattedTextarea` 和预览模板已支持 HTML 列表渲染，继续沿用。

测试：

- 增加/更新 `ResumeAiServiceImpl` 相关单元测试。
- 覆盖亮点标签清理、描述列表化、低质量评分封顶等关键规则。

## 验收标准

- 教育经历优化不会生成大段工作描述式亮点标签。
- 工作经历优化后的 `description` 是分点 HTML 列表。
- 项目经历优化后的 `description` 是分点 HTML 列表，并体现个人职责、技术实现、结果。
- 低质量简历不会得到 80+ 高分。
- 对缺少项目细节、缺少量化结果、描述不分点的简历，评分建议能指出具体模块问题。
