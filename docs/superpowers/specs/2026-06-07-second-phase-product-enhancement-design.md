# 面试吧二期产品增强设计

> 状态：产品方案级设计完成，待用户审阅
> 日期：2026-06-07
> 范围：简历模块增强、面试模块增强、职位模块增强、数据与分析模块

## 1. 背景与目标

面试吧当前已经完成用户、简历、AI 简历、语音面试、职位情报和首页统计等核心模块。二期目标不是继续堆叠孤立功能，而是把现有模块串成完整的程序员求职训练闭环。

二期产品定位：将面试吧升级为 **AI 程序员求职训练闭环平台**。

核心闭环：

```text
简历导入/优化 -> 目标职位导入 -> 岗位匹配分析 -> 针对性面试训练 -> 报告复盘 -> 能力成长分析 -> 反向优化简历
```

二期设计重点：

- 降低用户从旧简历进入系统的成本。
- 让简历优化、职位分析、模拟面试围绕同一个目标岗位联动。
- 将面试报告和岗位差距沉淀为可持续追踪的能力画像。
- 优先补齐求职闭环，不优先做商业化和大规模运营能力。

## 2. 总体用户路径

1. 用户上传旧简历 PDF/DOCX，系统解析为结构化简历模块。
2. 用户导入或选择目标职位，系统解析 JD、技术栈、隐性要求和风险点。
3. 系统对比简历和职位，输出匹配分、技能缺口、项目表达缺口。
4. 用户执行岗位定向整简历优化，生成更适合该岗位的简历版本。
5. 用户进入岗位定向模拟面试，选择面试类型、难度和时长。
6. 面试结束后获得评分、优秀回答示例、能力缺口和改进建议。
7. 数据分析模块更新能力雷达、成长趋势和下一步行动清单。
8. 用户根据建议继续优化简历、补齐技能、再次训练。

## 3. 简历模块增强

### 3.1 目标

让用户不只是创建简历，而是持续产出匹配目标岗位的高质量简历。

### 3.2 新增功能

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 简历文件导入 | 支持 PDF/DOCX 上传，AI 解析为结构化简历模块 | P0 |
| 一键整简历优化 | 按目标岗位整体润色简历，而不是只优化单个模块 | P0 |
| 岗位定向简历 | 从职位详情页生成针对该岗位的简历优化建议 | P0 |
| 简历版本 Diff | 对比两个版本的标题、模板、模块内容变化 | P1 |
| 版本回滚 | 从历史版本恢复当前简历 | P1 |
| 简历质量雷达 | 展示完整性、专业性、岗位匹配度、项目表达、关键词覆盖 | P1 |
| 简历分享链接 | 生成只读预览链接，用于投递或分享 | P2 |

### 3.3 页面设计

| 页面 | 改造点 |
|------|--------|
| `/resume` | 增加上传简历解析、基于岗位优化入口 |
| `/resume/:id/edit` | 增加整份优化、版本对比、回滚版本 |
| `/resume/:id/preview` | 增加分享链接、岗位匹配关键词覆盖 |
| `/resume/:id/diff` | 新增简历版本对比页 |

### 3.4 核心流程

1. 用户上传旧简历文件。
2. 系统提取文本内容并调用 AI 解析为标准 section 数据。
3. 用户确认解析结果并保存为新简历。
4. 用户选择目标职位进行整份优化。
5. AI 输出整体建议和分模块修改建议。
6. 用户可一键应用、逐条接受或忽略建议。
7. 系统保存新版本，支持 Diff 和回滚。

### 3.5 不包含能力

- 不做复杂在线排版设计器。
- 不做第三方招聘平台简历同步。
- 不做多语言简历自动翻译。

## 4. 面试模块增强

### 4.1 目标

让模拟面试从一次问答升级为有目标、有时间、有复盘、有成长轨迹的训练系统。

### 4.2 新增功能

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 面试计时器 | 支持总时长、单题答题时长、超时提示 | P0 |
| 面试中断恢复 | WebSocket 断开或页面刷新后恢复当前会话 | P0 |
| 岗位定向面试 | 基于目标职位和简历生成定制问题 | P0 |
| 面试类型扩展 | 技术面、项目深挖、HR 面、系统设计面 | P1 |
| 参考答案/优秀回答 | 报告中给出改进示例，而不只是建议 | P1 |
| 面试报告对比 | 对比多次报告，展示分数变化 | P1 |
| 追问策略增强 | 根据回答质量动态决定追问深度 | P1 |
| 面试录音回放 | 可选保存用户回答音频，用于复盘 | P2 |

### 4.3 页面设计

| 页面 | 改造点 |
|------|--------|
| `/interview/new` | 增加面试类型、目标职位、面试时长、难度选择 |
| `/interview/:id/room` | 增加计时器、断线恢复提示、题目进度 |
| `/interview/:id/report` | 增加优秀回答示例、历史对比、能力雷达 |
| `/interview/analytics` | 新增面试成长趋势页 |

### 4.4 核心流程

1. 用户选择简历和目标职位。
2. 用户选择面试类型、难度和时长。
3. 系统生成面试计划，包括题目数量、预计时长、考察点。
4. 面试房间显示题目进度、单题计时和总计时。
5. 页面刷新或 WebSocket 断开后，系统恢复当前会话状态。
6. 面试完成后生成报告、优秀回答示例、薄弱点和下一步练习建议。
7. 报告数据进入数据分析模块。

### 4.5 不包含能力

- 不做真人面试官排班。
- 不做实时多人视频面试。
- 面试录音默认关闭，只有用户主动开启时保存。

## 5. 职位模块增强

### 5.1 目标

让职位模块从 JD 解析工具升级为求职决策中心。

### 5.2 新增功能

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 职位搜索/列表 | 用户可查看已导入职位，按城市、技术栈、薪资筛选 | P0 |
| JD 关键词提取 | 提取硬技能、软技能、加分项、隐性要求 | P0 |
| 简历差距分析 | 对比用户简历与 JD，列出缺失技能和项目表达缺口 | P0 |
| 岗位面试题预测 | 基于 JD、公司、技术栈生成高概率面试题 | P0 |
| 批量导入职位 | 一次导入多个 JD URL | P1 |
| 投递状态管理 | 收藏、准备中、已投递、面试中、已拒绝、已 Offer | P1 |
| 职位风险识别 | 识别外包、996、薪资虚高、岗位模糊等风险 | P1 |
| 公司情报增强 | 融合资质、融资、舆情、招聘活跃度 | P2 |

### 5.3 页面设计

| 页面 | 改造点 |
|------|--------|
| `/job/import` | 增加批量导入 |
| `/job/favorites` | 升级为职位看板，支持投递状态 |
| `/job/:id` | 增加面试题预测、简历差距、投递状态 |
| `/job/list` | 新增职位列表/搜索/筛选 |
| `/job/:id/questions` | 新增岗位预测题库页 |

### 5.4 核心流程

1. 用户导入或批量导入职位。
2. 系统解析 JD、公司信息、技术栈和隐性要求。
3. 系统给出匹配分、风险点、面试重点和关键词。
4. 用户查看简历差距和预测面试题。
5. 用户一键进入岗位定向简历优化或岗位定向面试。
6. 用户更新投递状态，数据进入求职漏斗。

### 5.5 不包含能力

- 不做自动投递。
- 不做大规模全网职位爬取。
- 不绕过招聘平台反爬限制。

## 6. 数据与分析模块

### 6.1 目标

把用户在简历、职位、面试中的行为沉淀成求职能力画像，并输出下一步行动建议。

### 6.2 新增功能

| 功能 | 说明 | 优先级 |
|------|------|--------|
| 求职仪表盘 | 展示简历数、职位数、面试数、平均分、成长趋势 | P0 |
| 能力雷达图 | 展示技术准确性、表达清晰度、项目深度、岗位匹配、系统设计 | P0 |
| 技能缺口分析 | 从 JD 和面试报告中汇总缺失技能 | P0 |
| 成长趋势 | 按周/月展示面试分数、答题时长、追问通过率变化 | P1 |
| 学习建议 | 根据弱项生成学习任务和复习建议 | P1 |
| 求职漏斗 | 已收藏、已准备、已投递、面试中、Offer 等状态统计 | P1 |
| 目标岗位画像 | 用户最常投岗位、城市、薪资、技术栈偏好 | P2 |

### 6.3 页面设计

| 页面 | 功能 |
|------|------|
| `/analytics` | 总览仪表盘 |
| `/analytics/skills` | 技能缺口与能力雷达 |
| `/analytics/interviews` | 面试表现趋势 |
| `/analytics/jobs` | 求职漏斗与职位状态 |
| `/analytics/actions` | AI 生成的下一步行动清单 |

### 6.4 数据来源

| 来源模块 | 数据 |
|----------|------|
| 简历模块 | 技能、项目、版本质量评分、关键词覆盖 |
| 职位模块 | 目标岗位、JD 技术栈、匹配缺口、风险点、投递状态 |
| 面试模块 | 报告评分、薄弱点、回答质量、追问结果、答题时长 |
| 用户模块 | 目标岗位、技术方向、工作年限、城市、求职状态 |

### 6.5 分析输出

系统需要输出三类结果：

| 类型 | 示例 |
|------|------|
| 能力画像 | 后端开发、Spring Boot 熟练，系统设计薄弱，项目表达偏泛 |
| 技能缺口 | Redis 高可用、MySQL 索引优化、分布式事务、消息队列 |
| 行动建议 | 优先优化项目经历第 2 条；练习 5 道 MySQL 索引题；针对收藏职位 A 做一次系统设计面试 |

### 6.6 不包含能力

- 不做复杂 BI 报表系统。
- 不做跨用户排行榜。
- 不做企业招聘方数据看板。

## 7. 阶段规划

### 7.1 Phase 1：闭环补齐

Phase 1 目标是让用户可以完整走通“导入简历 -> 分析职位 -> 优化简历 -> 定向面试 -> 查看基础分析”的主链路。

功能：

- 简历文件导入。
- 整份简历 AI 优化。
- 岗位定向面试。
- 面试计时器。
- 职位列表/搜索。
- JD 关键词和简历差距分析。
- 求职仪表盘基础版。

### 7.2 Phase 2：复盘增强

Phase 2 目标是增强用户复盘、对比和持续训练能力。

功能：

- 简历 Diff 和回滚。
- 面试报告对比。
- 参考答案/优秀回答。
- 面试类型扩展。
- 投递状态管理。
- 能力雷达和技能缺口分析。

### 7.3 Phase 3：高级能力

Phase 3 目标是增强分享、批量处理和长期成长能力。

功能：

- 简历分享链接。
- 批量导入职位。
- 面试录音回放。
- 公司情报增强。
- 学习建议。
- 求职漏斗。
- 目标岗位画像。

## 8. 成功标准

二期完成后，系统应满足以下标准：

- 用户可以从上传旧简历开始，而不必从空白表单开始。
- 用户可以围绕一个目标职位完成简历优化和模拟面试。
- 面试过程具备时间感、进度感和中断恢复能力。
- 面试报告不只是评分，还能给出优秀回答示例和下一步行动。
- 职位模块不只是收藏职位，还能帮助用户判断匹配度、风险和准备重点。
- 数据分析模块能汇总简历、职位、面试数据，形成能力画像和技能缺口。

## 9. 后续实施建议

后续如果进入实现阶段，建议先为 Phase 1 单独拆分实现计划。Phase 1 应优先复用现有表和服务能力，避免一次性引入过多新模型。涉及文件解析、AI 调用、面试计时和统计汇总的能力应分别设计接口边界，避免把复杂逻辑堆入 Controller。
